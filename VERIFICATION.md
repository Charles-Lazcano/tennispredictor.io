# How predictions are logged and verified

Every prediction goes through the same tamper-evident chain before it's
considered "final." This document explains each link, what it actually
proves, and what it doesn't.

## 1. The prediction is made and inserted into `predictions.db` (SQLite)

- `created_at` is a SQLite `CURRENT_TIMESTAMP` default - generated by the
  database itself at insert time, never passed in by application code.
  There is no code path that lets a prediction be backdated or typed in
  manually.
- The predictions table has a `UNIQUE(match_date, player_a_id, player_b_id)`
  constraint, so the same match can't be predicted twice.
- **The predictions row is never edited after creation.** When a match's
  real result becomes known, it's inserted as a new row in a *separate*
  `results` table, joined back to its prediction by `(match_date,
  player_a_id, player_b_id)`. The original prediction is untouched.

## 2. `prediction_type` - the honesty label

- **`true_forward`**: predicted before the match existed in *any* data
  feed we pull from (via the RapidAPI upcoming-fixtures schedule). This is
  the only type that can honestly be called "walk-forward."
- **`same_day_fast_turnaround`**: predicted the moment a completed match
  first appears in TennisMyLife's results feed. Useful for exercising the
  pipeline and sanity-checking the model, but the outcome already existed
  in the source data at prediction time - it is **never** counted as
  walk-forward accuracy, and the report/dashboard label it accordingly.

## 3. Daily batch export + SHA-256 hash

Once a day's predictions are inserted, they're exported as one JSON file:
`data/predictions/YYYY-MM-DD.json`. That file is SHA-256 hashed - the hash
is what actually gets timestamped, not the file itself, so the proof is
tiny regardless of how much data is in the batch.

## 4. OpenTimestamps anchoring (the "Bitcoin stamp")

The SHA-256 hash is submitted to **4 independent OpenTimestamps calendar
servers** (`alice`/`bob`.btc.calendar.opentimestamps.org, btc.calendar.catallaxy.com,
finney.calendar.eternitywall.com). Each one returns a small merkle-path
proof saying "we saw this exact hash at this time." Those four proofs are
merged into a single `.ots` file sitting next to the JSON export.

**What this actually proves:** that the SHA-256 hash existed no later than
when it was submitted - which means the predictions inside that JSON file
existed, unmodified, at that point in time. If the file were changed by
even one byte afterward, its hash would no longer match the one anchored
in the `.ots` proof.

**What it takes to become fully verifiable against Bitcoin:** the
calendar servers periodically batch up everything submitted to them and
write one aggregate hash into a real Bitcoin transaction. Once that
transaction is mined, the `.ots` proof can be "upgraded" to include the
actual Bitcoin block height - at that point, anyone (not just us) can
verify the timestamp independently by checking the Bitcoin blockchain
directly, with no need to trust the calendar servers or us. This
aggregation + mining process typically takes several hours, sometimes
into the next day - it is normal for a same-day proof to still show
"pending" when checked.

The dashboard's **Verification Chain** tab has a "Check Bitcoin
confirmation" button per day, which re-queries the calendar servers and
reports whether that day's proof has been confirmed yet.

*(Implementation note: the official `ots` command-line tool is broken on
this Windows machine - its `python-bitcoinlib` dependency can't load a
compatible OpenSSL library. Rather than route around that, `pipeline/
ots_stamp.py` calls the underlying `opentimestamps` Python package's own
calendar-submission API directly - the `.ots` files it produces are
genuine, spec-compliant proofs, verified by round-tripping them through
the library's own deserializer.)*

## 5. Local git commit

Each day's prediction export, its `.ots` proof, the append-only scan log,
and the updated dataset are committed to the local git repository - a
second, independent record of "this existed as of this commit," backed by
git's own commit hashes and timestamps. **This currently stays local only
- nothing is pushed to a remote yet.** Once pushed to a public GitHub repo,
the commit history itself becomes a third, publicly-visible timeline
alongside the OpenTimestamps proof.

## 6. The append-only scan log

`logs/scan_log.jsonl` gets one line appended per run - timestamp, SHA-256
of every file fetched that day, row counts, and (once generated) the
prediction batch's hash and `.ots` proof path. The file is only ever
opened in append mode; there is no code path that rewrites a previous
entry.

## Summary: what would it take to fake a prediction after the fact?

You'd need to: rewrite a SQLite row with a fabricated `created_at` (not
possible through the app's own insert path, which never sets it),
regenerate a JSON export with a different hash, get *that* hash
independently timestamped earlier than the original (not possible - you
can't submit a hash to OpenTimestamps calendars in the past), and rewrite
git history convincingly on every clone that's ever pulled the repo. The
whole point of the chain is that each link is independently checkable
without trusting the previous one.
